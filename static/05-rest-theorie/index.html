* Die Begriffe Webanwendung und Webservice klingen zwar ähnlich, meinen aber unterschiedliche Dinge. * Insbesondere darf man bei Webservices nur bedingt an Dienstleistungen im Web denken. * **Webanwendung:** Wird über HTTP aufgerufen und nutzt HTML/CSS/JavaScript zur Erzeugung der Benutzeroberfläche. * **Webservice:** Nutzt ebenfalls HTTP als Transportprotokoll, überträgt jedoch weitgehend nur Rohdaten darüber. * Moderne (clientseitige) Webanwendungen nutzen in der Regel Webservices Kommunikation mit dem Backend.

Im Gegensatz zu früher ist es heute üblich, dass viele Anwendungen und Apps auf Dienste im Internet zugreifen. Dies reicht vom einfachen Auslagern einzelner Funktionen bis zu Anwendungen, bei denen die komplette Anwendungslogik entfernt wurde und der verbleibende Teil lediglich die Benutzeroberfläche implementiert. * Clientseitige Webanwendungen * Mobile Applikationen * Desktop-Programme mit „Cloudfunktionen” * Smart TV Apps * … Im Falle von Webanwendungen ist es heute üblich, fachliche Logik und Oberfläche zu trennen, so dass derselbe Webservice auch für Mobile Clients und andere Benutzeroberflächen verwendet werden kann. Aber auch um B2B-Kontext oder zur Kommunikation mehrerer Anwendungen untereinander kommen REST-Webservices häufig zum Einsatz.
Ist eine Webservice dafür gemacht, von unterschiedlichen Anwendungen aufgerufen zu werden, spricht man heute auch von einer API. Gemeint ist damit ein **Application Programming Interface**, das seine Funktionen verschiedenen Anwendungen zur Verfügung stellt. Ursprünglich kommt dieser Begriff aus der Betriebssystementwicklung, wo er für die Systembibliotheken zur Nutzung der Betriebssystemfunktionen in eigenen Anwendungen steht: * **WIN32-API:** Interne API von Microsoft Windows zur Erstellung von Windows-Anwendungen * **Java API:** Die mitgelieferte und sehr umfangreiche Klassenbibliothek der Programmiersprache Java * **Android API:** Aus der Java API abgeleitete Schnittstellen zur Programmierung von Android Apps In der objektorientierten Programmierung wird auch die öffentliche Schnittstelle einer Klasse gelegentlich als API bezeichnet. Im Rahmen des **API-driven Business** gehen heute immer Firmen dazu über, ihre Produkte und Dienstleistungen per REST API zur Verfügung zu stellen. Durch die Integration der APIs in fremde Webseiten, Anwendungen und Mobile Apps öffnen Sie auf diese Weise einen zusätzlichen Vertriebskanal für ihre Leistungen.

    GET

    • Inhalte vom Server abrufen
    • Die Anfrage darf keinen Request Body enthalten
    • Der Server antwortet mit dem gewünschten Inhalt oder einer Fehlerbeschreibung

    POST

    • Daten zur Verarbeitung an den Server schicken
    • Wird häufig dazu verwende, neue Inhalte auf dem Server abzulegen
    • Der Server antwortet mit dem gespeicherten Inhalt oder einer Fehlerbeschreibung

    PUT

    • Wird häufig verwendet, um vorhandene Daten auf dem Server zu aktualisieren
    • Der Inhalt der Datei muss im Request Body mitgeschickt werden
    • Der Server antwortet mit dem gespeicherten Inhalt oder einer Fehlerbeschreibung

    DELETE

    • Daten auf dem Server löschen
    • Die Anfrage darf keinen Request Body enthalten
    • Der Server antwortet mit einer Bestätigung oder einer Fehlberbeschreibung

    HTTP/0.9

    Dies war die ursprüngliche von Tim Berners-Lee entwickelte HTTP-Version, wie sie 1991 veröffentlicht wurde. In dieser Version gab es nur GET-Anfragen und noch keine Header Fields. Eine Anfrage bestand daher lediglich aus einer Zeile:

    GET /index.html

    Die Antwort des Servers beinhalte nur den Inhalt der abgerufenen Datei, sonst nichts.

    HTTP/1.0

    1996 wurde HTTP/1.0 veröffentlicht. Ab hier haben die HTTP-Nachrichten den oben gezeigten Aufbau.

    HTTP/1.1

    Diese Version wurde 1999 veröffentlicht und brachte weitreichende Verbesserungen. Unter anderem die Möglichkeit, die Verbindung zwischen zwei Anfragen offen zu halten. Denn zuvor wurde die Verbindung nach jeder Antwort automatisch getrennt. Da aber gerade der Verbindungsaufbau bei TCP/IP sehr langsam ist, führte dies zu langen Wartezeiten, wenn mehrere Dateien vom selben Server abgerufen werden mussten.

    HTTP/2

    2015 wurde HTTP/2 offiziell verabschiedet. In seinen Grundzügen basiert es auf dem von Google entwickelten SPDY-Protokoll. Die Inhalte der HTTP-Nachrichten sind weitgehend gleich geblieben, jedoch werden sie nicht mehr in der oben gezeigten Textform kodiert. Stattdessen kommt ein kompaktes und in vielen Dingen eindeutiger spezifiziertes Binärformat zum Einsatz.¹

    Als weitere größere Neuerung führt HTTP/2 Server Push ein, wodurch der Server beim Abruf einer HTML-Seite selbstständig weitere Dateien an den Browser schicken kann. Somit können Stylesheets, Skripte und Bilder schon an den Browser geschickt werden, bevor dieser überhaupt weiß, dass er sie benötigen wird.

    Stand 2020 hat sich HTTP/2 noch nicht vollständig gegen HTTP/1 durchgesetzt. Sehr viele Webseiten verwenden weiterhin HTTP/1. Inzwischen unterstützen aber alle Server, Browser und Frameworks das neue Protokoll.

    ¹ Beispielsweise war bei HTTP/1.x nie definiert, welche Bytefolge ein Zeilenende markiert.

    Angeleitet vom Konzept des **entfernten Methodenaufrufs** entwickelte sich Anfang der 2000er-Jahre **SOAP**, was ursprünglich für Simple Object Access Protocol stand, zu einem weit verbreiteten Standard. Der Webservice definiert hier einfach eine Reihe von Funktionen, die mit einer **HTTP-POST-Anfrage** entfernt aufgerufen werden können. Die aufgerufene URL ist für jede Funktion stets dieselbe, da der Name der Funktion sowie die für den Aufruf benötigten Parameter im HTTP Request Body als XML-Dokument übergeben werden. Analog dazu antwortet der Server mit einem ähnlich strukturierten XML-Dokument. Heute nimmt die Bedeutung von SOAP immer mehr ab, so dass SOAP in vielen Fällen durch REST verdrängt wurde. Die Grundidee eines Webservice-Aufrufs lässt sich daran aber immer noch gut erklären.
    Im Gegensatz zu SOAP orientiert sich REST vollständig an den Möglichkeiten von HTTP und überträge diese konzeptionell vom einfachen Abruf einer Webseite zur Verwaltung beliebiger „Ressourcen”. Ressourcen können in diesem Zusammenhang nahezu alle Dinge sein, sofern ihren Inhalt in einer geeigneten Form zwischen Client und Server ausgetauscht werden kann. Oft handelt es sich deshalb einfach um die Einträge innerhalb einer Datenbank. Aber auch IoT-Anwendungsfälle, bei denen jede Ressource für ein dingliches Objekt der realen Welt steht, sind denkbar, Die aufrufbaren Grundfunktionen entsprechen den unterschiedlichen HTTP-Verben, wodurch Ressourcen in REST angelegt, gelesen, überschrieben, verändert und gelöscht werden können. Die Abkürzung REST steht dabei für **Representational State Transfer**: * **Representational:** Oftmals unterstützen REST-Webservices mehrere Datenformate zur Codierung der Ressourcen. Standardformat ist meistens JSON, XML ist aber ebenfalls weit verbreitet. * **State Transfer:** Die Serverseite ist bei einem echten REST-Webservice komplett zustandslos und kann sich daher keine Informationen „temporär zwischenmerken”, ohne diese beispielsweise in einer Datenbank dauerhaft zu speichern. Bis auf wenige Ausnahmen beinhalten deshalb alle Anfragen und Antworten den kompletten Inhalt der angesprochenen Ressource. Innerhalb der URLs wird zwischen Ressourcen, Collections, Stores und Actions unterschieden

    Ein einzelner Datensatz wird Ressource genannt. Jede Ressource besitzt eine eindeutige URL, unter der sie aufgerufen werden kann.

    https://example.com/api/{resource}/{id}
    https://example.com/api/customers/47110815

    { "id": "47110815", "first_name": "Martina", "last_name": "Musterfrau", "registration_date": "2020-02-09" }
    GET PUT POST PATCH DELETE
    Ressource lesen Ressource überschreiben ––– Einzelne Werte ändern Ressource löschen

    Mehrere Datensätze werden in einer Collection gesammelt, ähnlich wie die Tabellen einer SQL-Datenbank.

    https://example.com/api/{collection}
    https://example.com/api/customers

    [ {"id": "47110815", "name": "Martina Musterfrau"}, {"id": "47110816", "name": "Toll Es Beispiel"}, {"id": "47110817", "name": "Klaus Kunde"} ]
    GET PUT POST PATCH DELETE
    Alle Ressourcen Lesen ––– Neue Ressource anlegen ––– Alle Ressourcen löschen

    Im Gegensatz zu Collections speichern Stores zwar ebenfalls mehrere Datensätze, die URL wird jedoch vom Client festgelegt.

    https://example.com/api/{collection}/{id}/{store}
    https://example.com/api/customers/47110815/favorites

    [ {"article": "9573387", "date_added": "2020-01-02"}, {"article": "5975741", "date_added": "2020-01-14"}, {"article": "1572246", "date_added": "2020-02-07"} ]
    GET PUT POST PATCH DELETE
    Inhalte auslesen Einträg hinzufügen ––– Einzelne Werte ändern Eintrag löschen

    Prozedurale Aktionen, die nicht dem üblichen Lesen und Schreiben entsprechen, werden Controller¹ oder auch Actions genannt.

    https://example.com/api/{collection}/{id}/{action}
    https://example.com/api/customers/47110815/checkout

    ¹ Der Name Controller ist unglücklich gewählt, weil die serverseitigen Klassen zur Implementierung eines REST-Webservices in den meisten Frameworks auch Controller genannt werden. In der Vorlesung verwenden wir daher lieber den Begriff Action.

    Jedes HTTP-Verb und jeder Stautscode besitzt in REST eine genau definierte Semantik. Bei einer Anfrage muss daher immer das passende HTTP-Verb gesendet werden. Ebenso muss der empfangene Statuscode ausgewertet werden, um den Erfolg der Anfrage zu prüfen.

    HTTP-Verben
    GET Daten lesen
    POST Daten unter einer vom Server generierten URL speichern
    PUT Daten unter einer bereits bekannten URL speichern
    PATCH Einzelne Felder eines vorhandenen Datensatzes ändern
    DELETE Daten löschen
    POST/PUT Aufruf einer prozeduralen Aktion

    GET Liste abrufen

    GET-Anfragen an einer Collection oder einen Store führen in der Regel dazu, dass eine Zusammenfassung aller enthaltenen Daten gesendet wird. Oftmals können Anzahl, Umfang und Sortierung der Daten über URL-Parameter gesteuert werden.

    HTTP-Anfrage

    GET /api/xfiles2018?sort_by=episode HTTP/1.1 Host: example.com Accept: application/json

    HTTP-Antwort

    HTTP/1.1 200 Ok Content-Type: application/json; charset=utf-8 [{ "episode": "2AYW01", "title": "My Struggle III" }, { "episode": "2AYW02", "title": "This" }, { "episode": "2AYW03", "title": "Plus One" }, { "episode": "2AYW04", "title": "The Lost Art of Forehead Sweat" }, { "episode": "2AYW05", "title": "Ghouli" }, { "episode": "2AYW06", "title": "Kitten" }, { "episode": "2AYW07", "title": "Rm9sbG93ZXJz" }, { "episode": "2AYW08", "title": "Familiar" }, { "episode": "2AYW09", "title": "Nothing Lasts Forever" }, { "episode": "2AYW10", "title": "My Struggle IV" }]
    GET Ressource abrufen

    GET-Anfragen an eine einzelne Ressourcen führen dazu, dass der komplette Datensatz übermittelt wird. Manche Webservices erlauben auch hier, über URL-Parameter nur bestimmte Felder anzufordern. Die meisten Webservices liefern aber einfach die komplette Ressource.

    HTTP-Anfrage

    GET /api/xfiles2018/2AYW04 HTTP/1.1 Host: example.com Accept: application/json

    HTTP-Antwort

    HTTP/1.1 200 Ok Content-Type: application/json; charset=utf-8 { "episode": "2AYW04", "title": "The Lost Art of Forehead Sweat", "directed_by": "Darin Morgan", "written_by": "Darin Morgan", "usa_air_date": "January 24, 2018", "de_air_date": "March 15, 2018", "running_time": 43, "guest_starts": [ "Brian Huskey", "Stuart Margolin" ] }
    POST Ressource anlegen

    POST-Anfragen werden in der Regel an eine Collection geschickt, um neue Daten anzulegen. Der Server vergibt daraufhin eine neue URL und teilt diese zusammen mit einer Kopie der gespeicherten Daten mit. Stellt der Server fest, dass der anzulegende Datensatz bereits vorhanden ist, kann er diesen entweder überschreiben oder dies mit dem Statuscode 409 Conflict ablehnen.

    HTTP-Anfrage

    POST /api/xfiles2018/2AYW04/reviews HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "star_rating": 10, "text": "One of the funniest episodes …" }

    HTTP-Antwort

    HTTP/1.1 201 Created Location: /api/xfiles2018/2AYW04/reviews/42 Content-Type: application/json; charset=utf-8 Cache-Control: no-cache { "id": 42, "episode": "2AYW04", "star_rating": 10, "text": "One of the funniest episodes …" }
    PUT Ressource aktualisieren

    Bei einer PUT-Anfrage kennt der Client die URL der abzuspeichernden Daten bereits, so dass die Anfrage direkt an die Ressource geschickt werden kann. Meistens wird dies zum Überschreiben vorhandener Datensätze genutzt. Es gibt aber auch Services, welche eine Neuanlage auf diese Weise erlauben. Manche Webservices liefern auch hier als Antwort eine Kopie der gespeicherten Daten. Üblich ist aber auch, einfach mit Status 204 No Content zu antworten und davon auszugehen, dass der Client due vollständigen Daten entweder bereits kennt oder mit einer nachfolgenden GET-Anfrage abrufen kann.

    HTTP-Anfrage

    PUT /api/xfiles2018/2AYW07 HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "episode": "2AYW07", "title": "Rm9sbG93ZXJz", "directed_by": "Glen Morgan", "written_by": "Shannon Hamblin & Kristen Cloke", "usa_air_date": "February 28, 2018", "de_air_date": "April 4, 2018", "running_time": 43, "guest_starts": [ "Candus Churchill" ] }

    HTTP-Antwort

    HTTP/1.1 204 No Content Location: /api/xfiles2018/2AYW07
    PATCH Einzelne Felder aktualisieren

    Soll ein vorhandener Datensatz aktualisiert werden, kann häufig eine PATCH-Anfrage gesendet werden, in welcher nur die zu überschreibenden Felder enthalten sind. Auf diese Weise wird unnötiger Traffic zwischen Client und Server vermieden.

    HTTP-Anfrage

    PATCH /api/xfiles2018/2AYW09 HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "directed_by": "James Wong", "written_by": "Karen Nielsen" }

    HTTP-Antwort

    HTTP/1.1 200 Ok Location: /api/xfiles2018/2AYW09 Content-Type: application/json; charset=utf-8 Cache-Control: no-cache { "episode": "2AYW09", "title": "Nothing Lasts Forever", "directed_by": "James Wong", "written_by": "Karen Nielsen", "usa_air_date": "March 14, 2018", "de_air_date": "April 11, 2018", "running_time": 44, "guest_starts": [ "Fiona Vroom", "Carlena Britch", "Jere Burns" ] }
    DELETE Ressource löschen

    Eine DELETE-Anfrage dient, wie der Name bereits sagt, dem Löschen von Daten. Sie wird daher in der Regel an eine einzelne Ressource geschickt, kann aber auch an eine Collection oder einen Store geschickt werden, um alle Daten auf einmal zu löschen. Als Antwort wird häufig nur eine Bestätigung ohne Dateninhalt geliefert.

    HTTP-Anfrage

    DELETE /api/reviews/13 HTTP/1.1 Host: example.com

    HTTP-Antwort

    HTTP/1.1 204 No Content
    PUT Aktion ausführen

    Prozedurale Aktionen werden durch eine PUT- oder POST-Anfrage ausgelöst, keinesfalls durch GET!¹ Viele Webservices verwenden POST, weil dies auch bei SOAP-Webservices (dem zweiten großen Standard neben REST) praktiziert wird, PUT wäre aber eigentlich logischer, da der Client die aufzurufende URL ja genau kennt.

    ¹ GET ist nur zum Abruf bereits vorhandener Daten gedacht und darf deshalb keine Seiteneffekte haben.

    HTTP-Anfrage

    PUT /api/xfiles2018/2AYW01/stream HTTP/1.1 Host: example.com Content-Type: application/json; charset=utf-8 Accept: application/json { "to_device": "smart_tv1", "start_at_timecode": 0, "audio": "english" }

    HTTP-Antwort

    HTTP/1.1 200 Ok Content-Type: application/json; charset=utf-8 { "stream_id": "2AYW01.smart_tv1.english" }

    Vollwertige REST-Webservices sollten immer **hypermedia driven** sein. Das heißt, sie sollten gemäß dem **HATEOAS-Prinzip** (Hypertext is the engine of the application state) mit jeder Antwort neben den Daten auch eine Liste mit Hyperlinks für alle zulässigen Folgeaktionen schicken. Als Beispiel zur Verdeutlichung diene eine einfache Onlinebanking API, bei der ein Konto folgende Zustände mit folgenden erlaubten Folgeaktionen haben kann:

    Es werden zwar die richtigen Daten zu einem Konto geliefert. Das Backend teilt aber nicht mit, wann welche Aktion erlaubt ist.

    [ { "number": 3699623, "type": "giro" }, { "number": 1013699623, "type": "savings" } ] { "account_number": 3699623, "bank_name": "Parkbank", "currency": "EUR", "amount": "2411.83", "state": "covered" }

    Durch Verlinkungen innerhalb der REST-Antworten teilt das Backend mit, wann welche Aktion erlaubt ist und wie diese aufzurufen ist.

    Es stehen sämtliche Links zum Tätigen von Überweisungen, Schließen des Kontos usw. zur Verfügung.

    { "account_number": 3699623, "bank_name": "Parkbank", "currency": "EUR", "amount": "2411.83", "state": "covered", "_links": { "self": { "href": "/account/3699623" }, "deposit": { "href": "/account/3699623/deposit" }, "pay": { "href": "/account/3699623/pay" }, "cash-transfer": { "href": "/account/3699623/transfer" }, "close": { "href": "/account/3699623/close" } } }

    Es stehen nicht alle Links zur Verfügung, da für ein überzogenes Konto nicht alle Aktionen erlaubt sind.

    { "account_number": 3699623, "bank_name": "Parkbank", "currency": "EUR", "amount": "0.00", "state": "not-covered", "_links": { "self": { "href": "/account/3699623" }, "deposit": { "href": "/account/3699623/deposit" } } }
